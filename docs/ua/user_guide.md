# Документація користувача

> Цей документ — для **користувача** системи (не розробника). Тут описано, що система вміє, як нею користуватися через веб-інтерфейс, які межі введення, скільки чекати, скільки коштує генерація і як читати помилки.
>
> Якщо ви розробник або хочете зрозуміти архітектуру — див. [`README.md`](../../README.md), [`system_overview.md`](./system_overview.md), [`system_overview_for_readers.md`](./system_overview_for_readers.md).

## 1. Що це і для кого

Система генерує, перевіряє та аналізує українські вірші класичної поетики (силабо-тонічна метрика). Підходить для:

- **Поетів** — швидко перевірити власний текст на метр і риму, побачити, де ритм збивається.
- **Викладачів і студентів-філологів** — наочна демонстрація стопи, клаузул, схем рим.
- **Дослідників** — вимірюваний пайплайн із RAG, фідбек-циклом, абляційними конфігураціями.
- **Просто допитливих** — згенерувати вірш на задану тему «під ключ».

## 2. Як запустити

Найпростіший шлях — Docker:

```bash
git clone https://github.com/DzOlha/poetry-generation-ua.git
cd poetry-generation-ua
echo "GEMINI_API_KEY=ваш_ключ_тут" > .env   # отримати на https://ai.google.dev
make serve                                   # http://localhost:8000
```

Без `GEMINI_API_KEY` система запуститься, але форма генерації буде **заблокована** з повідомленням «Не знайдено GEMINI_API_KEY». Перевірка / детекція / аналітика працюють без LLM-ключа.

Детальніше про змінні середовища — у [`README.md` § Environment Variables](../../README.md#environment-variables) і [`reliability_and_config.md`](./reliability_and_config.md).

### 3.1 Генерація

Поля форми:

| Поле | Тип | Межі | Дефолт |
|-------|-----|-------|--------|
| Тема | текст | 1–**200** символів | — |
| Метр | вибір зі списку | ямб / хорей / дактиль / амфібрахій / анапест | ямб |
| Стопи | число | 1–**6** | 4 |
| Схема рим | вибір | ABAB / AABB / ABBA / AAAA | ABAB |
| Строф | число | 1–**5** | 1 |
| Макс. ітерацій виправлення | число | 0–3 | 1 |

Що очікувати на виході: один або кілька катренів (4-рядкових строф) українською мовою з підсвіченими наголосами. Метрики в результаті: точність метру, точність рими, семантична релевантність до теми, успішність регенерації, кількість LLM-спроб, спожиті токени і вартість у $.

### 3.2 Перевірка

Поля: текст вірша (1–**5000** символів), очікуваний метр, кількість стоп (1–6), схема рим. На виході — посегментна підсвітка наголосів (очікувані / реальні / збіги), позначки порушень метру і неточних рим, % точності.

### 3.3 Детекція

Вхід — лише текст вірша (1–5000 символів) і чекбокси «що визначати» (метр / схему рим). Алгоритм перебирає 5 канонічних метрів × 1–6 стоп для метру і 4 канонічні схеми (ABAB / AABB / ABBA / AAAA) для рими. Поріг прийняття детекції — 85% точності для метру, ~50% для рими. **Не вміє** визначати: некласичні / силабічні / верлібр; XVIII ст. з нерівною акцентуацією; рідкісні діалектні форми (через слабкість стрес-резолвера).

### 3.4 Розширені конфігурації (абляції)

Сторінка для **дослідників**. 18 передзаданих сценаріїв (Normal / Edge / Corner) × 8 конфігів пайплайну (A–H):

- **A** — baseline: лише LLM + валідатор, без RAG, без feedback.
- **B–E** — нарощування з feedback-циклом: B (тільки feedback), C (RAG + feedback), D (metric examples + feedback), E — повна система.
- **F–H** — те ж саме що C/D/E, але **без feedback-циклу**: F = чистий RAG, G = чисті metric examples, H = обидва збагачення без feedback. Призначені для виміру **раw-внеску** компонентів у перший драфт (без маскування з боку feedback-репарації).

Запускає один сценарій через обраний конфіг, повертає трейс із метриками на кожній стадії пайплайну.

### 3.5 Аналітика якості

Дашборд із результатами абляційних прогонів (зазвичай 18 сценаріїв × 8 конфігів × 3 seeds = 432 запуски). Показує:

- Forest plot — внесок кожного компонента із 95% довірчими інтервалами;
- Box plot — розподіл метрик по конфігах;
- Heatmap — матриця сценарій × конфіг;
- Гістограми по категоріях.

На сторінці є розгортний **Глосарій термінів** (seed, paired delta, CI, Wilcoxon p-value тощо).

Звіт **не генерується автоматично** — потрібно запустити `make ablation` (432 LLM-виклики, ~2-3 години). На дефолтному `gemini-3.1-pro-preview` — ~\$50-100; на `gemini-2.5-pro` — ~\$35-70; на `gemini-2.5-flash` — ~\$3-7 (але якість абляції на Flash менш репрезентативна). Потім `make ablation-report`.

## 4. Обмеження введення (зведена таблиця)

Система валідує введення на трьох рівнях: HTML-форма, FastAPI Form / Query, Pydantic-схема. Усі межі однакові й відмовляють у запиті у разі їх недотримання.

| Поле | Min | Max | Чому така межа |
|-------|-----|-----|------------------|
| Тема (генерація) | 1 символ | 200 символів | Більше ніяк не покращить семантичний пошук, але роздує LLM-промпт |
| Текст вірша (валідація / детекція) | 1 символ | 5000 символів | ~50 рядків — практичний максимум для одного запиту; алгоритми лінійні, але час лінійно росте |
| Метр (вибір) | — | — | Лише 5 канонічних: ямб, хорей, дактиль, амфібрахій, анапест |
| Стоп у рядку | 1 | **6** | 7-8 стоп = 16-24 склади; рідкісне навіть для александрійського вірша |
| Схема рим (вибір) | — | — | Лише ABAB / AABB / ABBA / AAAA (катрени) |
| Строф | 1 | **5** | 5 × 4 = 20 рядків ≈ 1500 токенів output × 3 ітерації — практична межа |
| Макс. ітерацій виправлення | 0 | 3 | Понад 3 — лінійно росте вартість без помітного покращення (видно з абляцій) |

У разі порушення — отримаєте HTTP 422 (`Unprocessable Entity`) з повідомленням про конкретне поле.

## 5. Скільки часу і грошей

### Час

| Сценарій | Типово | Worst case |
|----------|--------|--------------|
| Перевірка / детекція (без LLM) | < 1 с | ~ 2-3 с (на 5000 символів) |
| Генерація 1 строфи, 1 ітерація | 30-60 с | ~ 2 хв |
| Генерація 1 строфи, 3 ітерації | 1-2 хв | ~ 4 хв |
| Генерація 5 строф, 3 ітерації | 2-3 хв | ~ 5-6 хв |

Worst case включає: один LLM-таймаут (120 с) + повторна спроба + експоненційний backoff. У формах це позначено як «Триває довше ніж очікувалось» після 2-х хвилин очікування.

**Перший запит після свіжого деплою** — додатково до 1-2 хв, бо система довантажує семантичну модель LaBSE (~1.8 ГБ) і Stanza-моделі для словника наголосів (~500 МБ). Подальші запити вже швидкі.

### Гроші

За замовчуванням використовується **Google Gemini** із моделлю **`gemini-3.1-pro-preview`** (визначається змінною `GEMINI_MODEL`). Це **платна** модель — інші моделі (зокрема Flash-родина) дають істотно гірші результати на українській поезії. Рекомендуємо саме pro-preview як baseline для типового використання.

#### Як налаштувати оплату

1. Перейдіть на <https://aistudio.google.com/apikey>, увійдіть із Google-акаунтом і створіть **API-ключ**.
2. Для pro-preview потрібно увімкнути billing на <https://aistudio.google.com/billing> — прив'язати кредитну картку.

   ```
   GEMINI_API_KEY=your_key_here
   ```

#### Якщо платити не хочете

Альтернатива — `gemini-2.5-flash` (вільне використання в межах free tier): орієнтовно **~10 запитів/хв, ~250/день** станом на квітень 2026 (Google періодично урізає ці ліміти — наприклад, у грудні 2025 вони впали на 50–80%). Перевіряйте актуальні значення на **<https://ai.google.dev/gemini-api/docs/rate-limits>**. **Якість для поезії помітно гірша** (плутає метр, рідше дотримує риму, частіше виробляє «несонгабельні» рядки) — годиться для smoke-тестування пайплайна, не для реальної генерації. Установіть:
```
GEMINI_MODEL=gemini-2.5-flash
```

**Якщо вичерпали ліміт** — отримаєте HTTP 429 і повідомлення «Досягнуто денного ліміту в N запитів». Лічильник скидається опівночі за тихоокеанським часом (PT).

## 6. Як читати метрики результату

| Метрика | Що означає | Що добре |
|----------|--------------|-------------|
| **Точність метру** | Частка рядків, що пройшли валідацію наголосу для очікуваного метру (з допуском до 2 жорстких помилок на рядок) | 100% — усі чисто; 75-99% — є кілька збоїв; <75% — серйозні розходження |
| **Точність рими** | Частка римованих пар, що пройшли фонетичний поріг (≥ 55% IPA-подібності від наголошеного голосного) | 100% — усі рими «справжні»; 50% — половина пар не зримувалась; 0% — рим немає взагалі |
| **Семантична релевантність** | Косинусна подібність LaBSE-ембедінгів теми та вірша, у % | ≥ 50% — вірш про задану тему; <30% — ймовірно не про неї |
| **Успішність регенерації** | Покращення сумарної оцінки між першою і останньою ітерацією, у пунктах | +X% — фідбек-цикл допоміг; 0% — без змін; від'ємне — погіршив (рідко, але буває) |
| **Спроб** | Кількість LLM-викликів: 1 — лише початковий драфт, +1 за кожну ітерацію виправлення | — |

На результатах можна навести курсор на іконку «ⓘ» поряд із метрикою, щоб прочитати детальне пояснення.

## 7. Помилки та як їх читати

| Код | Назва | Що означає | Що зробити |
|-----|-------|-------------|--------------|
| **422** | Validation Error | Введені дані не вкладаються в межі (наприклад, 7 стоп, 6 строф, тема порожня) | Виправити введення відповідно до меж із розділу 4 |
| **429** | Too Many Requests | Перевищено денний ліміт Gemini API | Зачекати до показаного часу (зазвичай ~22 год) або змінити ключ / модель |
| **502** | Bad Gateway | LLM не зміг згенерувати — мережева помилка, тимчасовий збій сервера Google, безпечні фільтри | Спробувати ще раз; якщо повторюється — змінити тему / параметри |
| **503** | Service Unavailable | LLM не сконфігурований (немає `GEMINI_API_KEY`) | Додати ключ у `.env` і перезапустити `make serve` |
| **500** | Internal Server Error | Несподіваний баг у системі | Перевірити логи (`docker compose logs`), завести issue |

Помилки користувача (422, 429) показуються українською, з конкретикою. Помилки інфраструктури (502, 503, 500) — короткі.

## 8. Чого система НЕ вміє

Чесний список:

1. **Силабічна / акцентна / верлібр поезія** — система калібрована під класичну силабо-тоніку. Сковорода / Шевченків Кобзар XVIII ст. з нерівною акцентуацією може бути визначений неправильно.
2. **Інші мови** — лише українська (стрес-словник, IPA-транскриптор, RAG-корпус — усі UA-only).
3. **Білий вірш / нерегулярна структура** — система **очікує** регулярний метр і одну з 4 схем рим (ABAB / AABB / ABBA / AAAA). Вільний вірш — не для цієї системи.
4. **Не-катренні строфи** — лише 4 рядки на строфу (терцети, секстини, сонети «3+3+4+4» — поки ні).
5. **Стресове наголошення на рідкісних / запозичених / неологізмах** — резолвер може помилитись, бо словник обмежений. Це впливає і на валідацію, і на детекцію.
6. **Реальний час / стримінг** — генерація атомарна, не побачиш «літери з'являються». Прогрес — лише через таймер.
7. **Голосовий вивід / TTS** — лише текст.

## 9. JSON API

Якщо потрібна програмна інтеграція — є JSON API. Документація: [`/docs`](http://localhost:8000/docs) (Swagger UI).

Основні ендпоінти:

| Метод | URL | Призначення |
|--------|-----|--------------|
| GET | `/health` | Health check |
| GET | `/system/llm-info` | Чи готовий LLM (для UI) |
| POST | `/poems/generate` | Згенерувати вірш |
| POST | `/poems/validate` | Перевірити вірш |
| POST | `/poems/detect` | Визначити метр / схему |

| POST | `/evaluation/run` | Запустити сценарій через конфіг |

Усі ендпоінти повертають JSON. Помилки — стандартні HTTP-коди (422 / 429 / 502 / 503 / 500), з тілом `{"error": "...", "type": "..."}`.

## 10. Часті питання

**Q: Чому моя тема дала вірш зовсім не про неї?**
A: Перевірте «Семантична релевантність» — якщо < 30%, LLM «змилив» убік. Спробуйте перефразувати тему конкретніше: «весна у лісі», а не просто «весна».

**Q: Метр 100%, але насправді ритм відверто збитий — чому?**
A: Стрес-резолвер може ставити наголоси не там, де поет хотів. На рідкісних словах це ймовірно. Найкращий тест — прочитати вголос.

**Q: Чому рима 50%, хоча всі рядки рифмуються?**
A: Фонетичний валідатор міряє IPA-подібність. Якщо у вас, наприклад, очна / суфіксальна рима («шибочках / кутиках» — спільне «-ках», але різні наголошені голосні) — алгоритм не зарахує. Це навмисно, щоб не пропускати псевдо-рими.

**Q: Чи можна прискорити генерацію?**
A: Зменшити «Макс. ітерацій» до 0 — отримаєте швидко, але без виправлень. Або обрати дешевшу модель (наприклад, `gemini-2.5-flash` замість дефолтного `gemini-3.1-pro-preview`) — у `.env` через `GEMINI_MODEL`. Зважайте, що Flash-моделі дають істотно гірші результати для поезії.

**Q: Чи зберігаються мої тексти / результати?**
A: Ні. Все в оперативній пам'яті, після перезапуску — все стирається. Окремо `make ablation` пише результати в `results/<timestamp>/`.

**Q: Безпека API-ключа — чи витікає він кудись?**
A: Ключ використовується лише для викликів до Google Gemini. Не пишеться в логи, не передається ні в які інші сервіси. Усі промпти ідуть напряму до Google.
