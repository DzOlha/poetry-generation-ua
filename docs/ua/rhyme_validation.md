# Валідація рими

> Алгоритм, який порівнює закінчення рядків **за звучанням**, а не за графікою. Критичний нюанс: рима — це фонетика, тому спочатку переклад у IPA, потім математична відстань між звуками.

## Лінгвістична основа правил

Правила перевірки рими в цій системі — **не власна вигадка**, а формалізація канонічних визначень української поетики. Якщо коротко: **рима = фонетична подібність від наголошеного голосного до кінця слова**, з допуском на типи (точна / асонанс / консонанс / неточна).

### Канонічне визначення

«Рима — звуковий повтор у клаузулах двох (і більше) рядків, що починається з наголошеного голосного й охоплює всі наступні звуки» — стандартне формулювання у:

- **І. Качуровський** (1967) «Фоніка» / «Строфіка».
- **М. Сидоренко** (1985) «Українське віршування».
- **Н. Костенко** (2006) «Українське віршування XX століття».

Ключові слова — **«звуковий», «наголошений голосний», «всі наступні звуки»**. Звідси випливають усі інші правила:

| Правило в системі | Що відображає в каноні |
|--------------------|--------------------------|
| Транскрипція в IPA | «звуковий повтор» — порівнюємо звучання, не графіку |
| Rhyme-part стартує з наголошеного голосного | «починається з наголошеного голосного» |
| Лівостороннє вирівнювання, повна довжина | «охоплює всі наступні звуки» — без обрізання |
| Класифікація EXACT / ASSONANCE / CONSONANCE / INEXACT | стандартна типологія рим у Качуровського |
| Stressed-vowel gate | без збігу опорного голосного — це не рима (виняток: дисонанс) |

### Чому саме чотири типи (EXACT / ASSONANCE / CONSONANCE / INEXACT)

Канонічна типологія сучасної української поетики:

- **Точна (EXACT)** — повний збіг звуків від наголошеного голосного («біль / ціль»).
- **Асонанс** — збіг лише голосних, приголосні різні («дивиться / висіти»).
- **Консонанс / дисонанс** — збіг приголосних, голосні різні («гра́д / звід», «по́лем / до́лом»).
- **Неточна** — є спільність, але ні голосні, ні приголосні не дотягують до 75% збігу.

Такий поділ присутній у класиків від Качуровського до сучасних посібників. Алгоритм у [`_classify_precision`](../../src/infrastructure/validators/rhyme/pair_analyzer.py) — пряма реалізація цих визначень: він рахує окремо vowel-channel і consonant-channel similarity, а потім класифікує за порогом 0.75.

### Чому консонанс важливіший за повну строгість

Підручники прямо говорять: **сучасна українська поезія активно використовує неточні рими** (асонанс, консонанс), і це не дефект, а виразний засіб (Шевченко, Костенко, Антонич). Тому валідатор не вимагає 100% звукового збігу — він приймає пару як риму, якщо score ≥ 0.55 І класифікація ≠ NONE.

### Що НЕ враховано

Канон згадує також **глибокі рими, опорні рими, банальні рими** — їх класифікація потребує контекстного аналізу всього вірша і виходить за межі цього алгоритму. Ми обмежуємось перевіркою, що пара римується **взагалі**, не оцінюємо її художню вартість.

## Чому фонетика, а не буквенне порівняння

Класичний приклад:
- «**біль**» / «**ціль**» — різне написання букв на кінці, але звучить однаково `[b⁽ʲ⁾ilʲ]` / `[tsʲilʲ]` — **справжня** рима.
- «**вода**» / «**сода**» — на око схоже, звучання `[voˈda]` / `[ˈsɔda]` — наголоси в різних місцях, **не** справжня рима (рима ослаблена).
- «**спить**» / «**спить**» — однакове написання і звучання — **тавтологічна** рима (насправді не рима, а повтор слова).

Отже: графіка → оманлива. Нам потрібна фонетична транскрипція. В українській це переважно чиста трансляція кирилиці у IPA (Міжнародний фонетичний алфавіт) за простими правилами.

## Загальний потік

```
Поема              →  Схема "ABAB"  →  пари рядків [(0,2), (1,3)]
   │
   └─ для кожної пари:
        │
        ├─ взяти останнє слово обох рядків
        ├─ резолвити наголос обох
        ├─ виокремити римувальну частину (від наголошеного голосного до кінця, IPA)
        ├─ перевірка опорного голосного (gate): різні наголошені голосні допускаються
        │   лише за збігу приголосних наголошеного складу (консонанс)
        ├─ Левенштейн на фонемах (від наголошеного голосного, без суфіксального обрізання)
        ├─ класифікувати клаузулу (MASCULINE / FEMININE / DACTYLIC / HYPERDACTYLIC)
        ├─ класифікувати точність (EXACT / ASSONANCE / CONSONANCE / INEXACT / NONE)
        └─ визначення is_valid (similarity ≥ threshold)
```

Уся фонетична частина інкапсульована у [`PhoneticRhymePairAnalyzer`](../../src/infrastructure/validators/rhyme/pair_analyzer.py) (реалізація `IRhymePairAnalyzer`). Він повертає `RhymePairAnalysis` — value object із усіма діагностичними полями (`rhyme_part_a/b`, `score`, `clausula_a/b`, `precision`). [`PhoneticRhymeValidator`](../../src/infrastructure/validators/rhyme/phonetic_validator.py) лише оркеструє: бере splitter, tokenizer, scheme extractor, pair analyzer і поріг — і збирає `RhymeResult` зі структурним `PairFeedback` на кожну невдалу пару.

## 1. Виокремлення пар зі схеми

[`StandardRhymeSchemeExtractor`](../../src/infrastructure/validators/rhyme/scheme_extractor.py) бере рядок схеми і кількість рядків у строфі. Повертає `list[tuple[int, int]]` — пари 0-based індексів рядків, що мусять римуватись.

Підтримувані схеми (4 рядки на строфу):

| Схема | Пари у строфі | Приклад |
|-------|---------------|---------|
| ABAB | (0, 2), (1, 3) | Перехресна рима |
| AABB | (0, 1), (2, 3) | Сусідня (парна) |
| ABBA | (0, 3), (1, 2) | Обіймаюча |
| AAAA | (0, 1), (0, 2), (0, 3), (1, 2), (1, 3), (2, 3) | Монорима — всі попарно |

Для кількох строф шаблон повторюється зі зсувом: у вірші з 8 рядків і схемою ABAB виходить `[(0,2), (1,3), (4,6), (5,7)]`.

Алгоритм парсеру:
1. Розбити схему на літери (*A, B, A, B*).
2. Згрупувати рядки за літерою: *{A: [0, 2], B: [1, 3]}*.
3. Згенерувати всі **комбінації пар** у кожній групі (не перестановки — (0,2) == (2,0)).
4. Відсортувати → повторити по строфах.
5. Порожня схема або повністю унікальні літери (як «ABCD») → `UnsupportedConfigError` українською.

## 2. Клаузула — що саме рівняємо

**Клаузула** — суфікс рядка від **наголошеного складу останнього слова до кінця рядка**. Усе, що стоїть до наголошеного складу, — не частина рими.

Наприклад, рядок *«Де тихо світяться вогні́»*:
- Останнє слово: «вогні́», наголос на «і́» (позиція 1, бо 2 склади: во-гні).
- Клаузула: `гні́` (1 склад — **чоловіча клаузула**).

Рядок *«Ворушаться тіні чужо́го»*:
- Останнє слово «чужо́го», наголос на «о́» у «-жо-» (позиція 1 з 3).
- Клаузула: `жо́го` (2 склади — **жіноча**).

Класифікація клаузули — enum [`ClausulaType`](../../src/domain/value_objects.py) — за кількістю ненаголошених після наголосу:

| Enum | Ненаголошених після | Приклад |
|------|---------------------|---------|
| `MASCULINE` (чоловіча) | 0 | бі́ль, сві́т |
| `FEMININE` (жіноча) | 1 | ходо́к, кни́га |
| `DACTYLIC` (дактилічна) | 2 | ро́зум, молодо́го |
| `HYPERDACTYLIC` (гіпердактилічна) | 3+ | розумі́ється |
| `UNKNOWN` | — | порожній / неопрацьовуваний вхід |

Реалізація: [`PhoneticRhymePairAnalyzer._detect_clausula(word)`](../../src/infrastructure/validators/rhyme/pair_analyzer.py) — рахує склади через ін'єктований `ISyllableCounter` і шукає наголошений склад через `IStressResolver`. Сам "rhyme part" будує [`UkrainianIpaTranscriber.rhyme_part(word, stress_idx)`](../../src/infrastructure/phonetics/ukrainian_ipa_transcriber.py).

## 3. Транскрипція у IPA

[`UkrainianIpaTranscriber`](../../src/infrastructure/phonetics/ukrainian_ipa_transcriber.py) перетворює кириличний рядок у IPA-нотацію. Спрощена таблиця:

```
а → a    о → ɔ   я → ʲa / ja   '  → ʲ (м'якість)
е → ɛ    у → u   ю → ʲu / ju   ь  → ʲ (м'якість)
и → ɪ    і → i   є → ʲɛ / jɛ   ґ  → ɡ
е (ненагол.) → ɛ або ɘ  (спрощено як ɛ)
...
```

Важливі правила:
- **Йотування.** `я`, `ю`, `є`, `ї` після голосної або на початку слова дають **два звуки**: `ja`, `ju`, `jɛ`, `ji`. Після приголосної — `ʲa`, `ʲu`, `ʲɛ`, `i` (з попередньою мʼякістю).
- **Мʼякий знак `ь`** транскрибується як мʼякість попередньої приголосної `[ʲ]`, не окремий звук.
- **Дзвінкі/глухі приголосні** — у кінці слова НЕ оглушуються (українська зберігає дзвінкість, на відміну від російської). Тому «сад» = `[sad]`, не `[sat]`.

Приклади:
```
"сон"    → "sɔn"
"вогні"  → "vɔɡnʲi"
"тіні"   → "tʲini"
"мрія"   → "mrʲija" / "mrʲiʲa" (сучасна норма — з j)
"країни" → "krajinɪ"
```

## 4. Перевірка опорного голосного (gate)

[`PhoneticRhymePairAnalyzer._stressed_syllables_align(r_a, r_b)`](../../src/infrastructure/validators/rhyme/pair_analyzer.py):

Канонічне правило української рими — це **тотожність опорного (наголошеного) голосного** плюс подібність наступних звуків. Перш ніж рахувати Левенштейн, перевіряємо умову прийнятності пари:

| Випадок | Дія |
|---------|-----|
| Наголошені голосні **збігаються** | Пара йде далі на скоринг (точна / асонансна / неточна рима) |
| Наголошені голосні **різні**, але приголосні наголошеного складу мають similarity ≥ 0.75 | Пара йде далі (це варіант **консонансу**: «по́лем / до́лом», «гра́д / звід») |
| Наголошені голосні **різні** і приголосні наголошеного складу теж різні | Пара відхиляється (`score = 0`, `precision = NONE`) — слова мають лише спільне граматичне закінчення (наприклад, «шибочка́х / кутика́х» — спільний суфікс «-ках», але різні наголошені склади) |

«Приголосні наголошеного складу» — IPA-приголосні між наголошеним голосним і наступним голосним; для чоловічої клаузули — увесь пост-стресовий хвіст.

## 5. Нормована відстань Левенштейна

[`PhoneticRhymePairAnalyzer._suffix_aligned_score(r_a, r_b)`](../../src/infrastructure/validators/rhyme/pair_analyzer.py):

1. **Лівостороннє вирівнювання.** Обидві IPA-частини вже починаються з наголошеного голосного, тож порівнюються від початку без обрізання. Різниця довжин (наприклад, 8-символьна римована частина «ши́бочках» проти 4-символьної «ку́тиках») природно знижує оцінку — це і є канонічна вимога: пост-стресові послідовності мають збігатися.
2. **Левенштейн** — мінімальна кількість вставок / видалень / замін (через ін'єктований порт `IStringSimilarity`).
3. **Нормалізація:** `similarity = 1 - (distance / max(len_a, len_b))`.
4. Повертається число у `[0, 1]`, де 1 = ідентично.

Окремо аналізатор пробує **резолвлений наголос** *і* **передостанній склад** (якщо вони не збігаються) і обирає пару кандидатів, що дає найвищу оцінку після проходження gate — захист від шуму резолвера на рідкісних словах.

Реалізація Левенштейна — у [`src/shared/string_distance.py`](../../src/shared/string_distance.py), обгортка як порт `IStringSimilarity`.

## 6. Класифікація рими

Enum [`RhymePrecision`](../../src/domain/value_objects.py) маркує кожну пару. [`PhoneticRhymePairAnalyzer._classify_precision(rhyme_a, rhyme_b, overall_score)`](../../src/infrastructure/validators/rhyme/pair_analyzer.py):

```
overall_score ≥ 0.95  → EXACT        (точна рима: "біль" / "ціль")

else:
  обрізати обидві частини до довжини коротшої (suffix alignment)
  vowels_X     = [c for c in trimmed_X if c in "aeiouɪ"]
  consonants_X = [c for c in trimmed_X if c not in vowels]

  vow_sim = similarity(vowels_a, vowels_b)
  con_sim = similarity(consonants_a, consonants_b)

  vow_sim ≥ 0.75 AND con_sim < 0.75   → ASSONANCE
  con_sim ≥ 0.75 AND vow_sim < 0.75   → CONSONANCE
  overall_score > 0                    → INEXACT
  overall_score == 0                   → NONE
```

Інтерпретація:
- **EXACT** — словникова рима («біль / ціль», «день / пень»).
- **ASSONANCE** — рима голосних: «дивиться» / «висіти» (і-и-и… і-і-и…) — голосні збігаються, приголосні ні.
- **CONSONANCE** — рима приголосних: «полем» / «долом» (п-л-м / д-л-м) — приголосні збігаються, голосні різні.
- **INEXACT** — є спільність, але ні голосні, ні приголосні не дотягують до 75%.
- **NONE** — нічого спільного.

Усі типи крім NONE **зараховуються як римовані**, якщо `score ≥ 0.55` (поріг валідності). Класифікований тип зберігається у `RhymePairAnalysis.precision` і пропускається у `PairFeedback.precision` для UI та feedback-у LLM.

## 7. Агрегація у вірш

[`PhoneticRhymeValidator.validate(poem_text, rhyme_scheme)`](../../src/infrastructure/validators/rhyme/phonetic_validator.py):

1. Розбити вірш на рядки через `ILineSplitter`.
2. Витягти пари через `IRhymeSchemeExtractor` ([`StandardRhymeSchemeExtractor`](../../src/infrastructure/validators/rhyme/scheme_extractor.py)).
3. Для кожної пари викликати `IRhymePairAnalyzer.analyze(...)` → `RhymePairAnalysis`, далі обгорнути в `RhymePairResult(ok=score ≥ threshold, ...)`.
4. Агрегувати:
   - `ok = all(pair.ok for pair in pair_results)`.
   - `accuracy = valid_pairs / total_pairs` (або `1.0` якщо пар немає).
   - Зібрати [`PairFeedback`](../../src/domain/models/feedback.py) (зверніть увагу: структурний DTO тепер у `src.domain.models.feedback`, перенесений з `src.domain.feedback`) на кожну невалідну пару. `IFeedbackFormatter` рендерить його у текстовий рядок, який бачить LLM.

## Тонкощі і обмеження

- **Неідеальна IPA-транскрипція.** Правила спрощені: не враховуємо асиміляцію (*вокзал* = `[voɡzal]` насправді, але транскриптор дає `[voksal]`), редукцію ненаголошених `[o → ɔ̯]` тощо. Для порівняльної мети досить.
- **Наголос — слабка ланка.** Якщо Stressifier поставив наголос не там, де поет мав на увазі, клаузула виокремиться криво → рима може проскочити як валідна або навпаки. Особливо бʼє по рідкісних і запозичених словах.
- **Монорими AAAA.** 6 пар на строфу замість 2 — генерація має складніше завдання, і навіть одна помилкова пара різко знижує accuracy.
- **Поріг 0.55** — експериментально підібраний. На lower — пропускає випадкові співпадіння як рими, на higher — не приймає класичні неточні рими.
- **Лівостороннє порівняння без обрізання** — після переходу на повну довжину римованих частин слова з різною глибиною наголосу (наприклад, дактилічна «ши́бочках» vs жіноча «селу́») природно отримують нижчу оцінку. Це бажана поведінка: справжня рима вимагає схожої довжини пост-стресової послідовності.
